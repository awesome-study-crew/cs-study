# 키(key)

- 테이블의 레코드를 식별할 수 있는 하나 이상의 필드를 `키(key)` 라고 한다.

## 사용할 스키마

### 📘 Students (학생)

| Column Name | Data Type    | Nullable | Key    | Description           |
| ----------- | ------------ | -------- | ------ | --------------------- |
| student_id  | VARCHAR(20)  | NO       | PK     | 학번 (기본키)         |
| rrn         | VARCHAR(20)  | NO       | UNIQUE | 주민등록번호 (대체키) |
| email       | VARCHAR(100) | NO       | UNIQUE | 이메일 (대체키)       |
| name        | VARCHAR(50)  | NO       |        | 학생 이름             |
| dept        | VARCHAR(50)  | NO       |        | 학과                  |

#### 📌 Key Summary

- Primary Key: `student_id`
- Candidate Keys: `student_id`, `rrn`, `email`
- Alternate Keys: `rrn`, `email`
- Example Super Keys:
  - `{student_id}`
  - `{student_id, name}`
  - `{email}`

### 📘 Enrollments (수강신청)

| Column Name | Data Type   | Nullable | Key    | Description          |
| ----------- | ----------- | -------- | ------ | -------------------- |
| student_id  | VARCHAR(20) | NO       | PK, FK | 학번 (Students 참조) |
| course_id   | VARCHAR(20) | NO       | PK     | 과목 ID              |
| semester    | VARCHAR(10) | NO       | PK     | 학기 (예: 2026-1)    |
| grade       | CHAR(2)     | YES      |        | 성적                 |

#### 📌 Key Summary

- Composite Primary Key: `{student_id, course_id, semester}`
- Foreign Key:
  - `student_id` → `Students(student_id)`
- Example Super Keys:
  - `{student_id, course_id, semester}`
  - `{student_id, course_id, semester, grade}`

---

## 슈퍼키(Super Key)

> [!IMPORTANT]
>
> - 하나의 튜플(행)을 유일하게 식별할 수 있는 컬럼들의 집합
> - 참고로, 유일성만 만족하면 슈퍼키이다. 불필요한 컬럼이 끼어도 괜찮다.

- `Students` 테이블에서 `student_id`, `rrn`, `email` 등으로 식별할 수 있다. 이는 모두 슈퍼키이다.
- `{student_id, name}` 과 같이 이미 `student_id`로 유일한데, `name`이 불필요하게 추가되어도 슈퍼키에 해당한다.

## 후보키(Candidate Key)

> [!IMPORTANT]
>
> - 슈퍼키 중에서 최소성까지 만족하는 키
> - 즉, 유일성과 최소성을 모두 만족하는 키

- 슈퍼키에서는 가능했던 `{student_id, name}` 집합이 후보키에서는 불가능하다.
- `name` 속성이 없어도 `student_id`로 구별이 가능하기 때문에 최소성을 만족하지 않는다. 따라서 후보키가 될 수 없다.

## 복합키(Composite Key)

> [!IMPORTANT]
>
> - 2개 이상의 컬럼을 묶어서 키(식별자) 역할을 하는 것
> - 참고로, 복합키는 슈퍼키/후보키/기본키 어디든 될 수 있다. 단순히 구성 컬럼 수를 말하는 개념이다.

- `Enrollments` 테이블에서 수강신청을 유일하게 만들려면 `{student_id, course_id, semester}` 가 키여야 한다.
- 이를 기본키로 채택하면 `복합 기본키`가 되는 것이다.

## 기본키(Primary Key)

> [!IMPORTANT]
>
> - 후보키들 중에서 데이터베이스 설계자가 대표로 선택한 키 1개
> - NULL 불가, 중복 불가 등의 특징이 있다.

- `Students` 테이블에서는 위에서 언급한 후보키 모두 기본키가 될 수 있다.
- 데이터베이스 설계자인 필자가 `student_id` 후보키로 선택하면 `student_id`가 기본키가 되는 것이다.

## 대체키(Alternate Key)

> [!IMPORTANT]
>
> - 후보키 중에서 기본키로 선택되지 않은 나머지 후보키

## 외래키(Foreign Key)

> [!IMPORTANT]
>
> - 한 테이블의 컬럼이 다른 테이블의 PK(또는 UNIQUE)를 참조해서 참조 무결성을 강제하는 키

- `Enrollments` 테이블에서 `student_id`는 `Students` 테이블의 `student_id` 컬럼을 참조할 수 있다.
- 이는 `Enrollments` 테이블에 `S999` 라는 `student_id`가 들어오려면 `Students` 테이블의 `student_id` 중에서 `S999` 인 학생이 있어야 한다.
- 물론 학생을 삭제할 때도 제약이 걸릴 수 있다 (ex. `ON DELETE CASCADE/RESTRICT`) 등
